# 詳細設計：バッチ仕様書

## 1. ■概要・目的

非同期・定期実行されるバッチ処理の詳細設計を行います。起動条件、データ取得・加工・出力のシーケンス、トランザクション制御、エラー発生時のリカバリ方法等を定義することで、バッチ処理の安定稼働とデータ整合性を確保します。

## 2. ■事前準備

- [ ] 規定のテンプレート：[仕様書_template.md](./仕様書_template.md)
- [ ] 規定のサンプル：[仕様書_sample.md](./仕様書_sample.md)
- [ ] 処理内容の概説、対象エンティティの洗い出し
- [ ] バッチ実行基盤の仕様確認

## 3. ■作業手順（Issueのタスクリストとして利用）

- [ ] **テンプレートの利用**：[仕様書_template.md](./仕様書_template.md) をコピーして作成を開始します。ファイル名は `バッチID_バッチ名.md` 形式とします。
- [ ] **基本情報の定義**：バッチID・バッチ名・実行タイミング・Cron式・処理種別を記載します。
- [ ] **目的・処理対象の記述**：バッチの目的、対象テーブル、抽出条件を記載します。
- [ ] **処理フローの設計**：入力データの読み取り、フィルタリング、計算、出力（DB更新／ファイル出力／メール送信）のステップを `### ■ ステップN：〇〇` で記載します。各ステップで発生しうるエラーはインデントして記載します。
- [ ] **入出力データ定義**：参照フィールド・更新フィールドをそれぞれ表で記載します。JOIN条件や特記事項は `▶ 備考` サブセクションに記載します。
- [ ] **エラーハンドリング**：エラー種別ごとに `### ■` で区切り、発生タイミング・対応内容を記載します。
- [ ] **運用設計**：ログ出力先、リカバリ手順を記載します。
- [ ] **全体備考の記述**：処理件数想定、チャンクサイズ、パフォーマンス面の配慮などを記載します。

## 4. ■セルフチェック項目（作業完了後の最終確認）

- [ ] 実行トリガー（Cron式・起動タイミング）、入出力対象が特定されている
- [ ] 処理フローが `### ■ ステップN` 形式で記載され、各ステップのエラーハンドリングが明示されている
- [ ] 更新トランザクションの単位（コミット／ロールバック）が定義されている
- [ ] エラーハンドリングに、発生タイミング・対応内容が記載されている
- [ ] パフォーマンスへの配慮（バッチサイズ・チャンク処理等）が行われている
- [ ] ログ出力先、異常終了時のリカバリ手順が明記されている
- [ ] 冪等性（同一引数での再実行可否）の方針が記載されている
- [ ] テンプレートの `> [記]` で始まる行をすべて削除している
- [ ] 不要な項目、および他ドキュメントから変更のない重複記載を避けている
- [ ] AIによる自動生成内容を人間がレビューし、必要に応じて修正・補足されている
- [ ] ユーザー（角田さん）、PMが見て処理内容やリカバリ手順を理解できる内容になっている
