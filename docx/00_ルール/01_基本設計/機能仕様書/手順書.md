# 基本設計：機能仕様書

## 1. ■概要・目的

セクションごとのテンプレートファイルを順番に埋めながら機能仕様書を作成します。各セクションの記述に必要になった時点でWF・シーケンス図・API概要・テーブル定義を作成し、機能仕様書に反映します。TMRVおよびPDRVのレビュー対象となる基本設計の最終成果物です。

### ■ 記述方針（重要）

機能仕様書は **概念レベルの業務仕様書** です。実装詳細（コード）と混ざらないよう、以下のルールを守ります。

- **原則、変数名・カラム名・クラス名・enum名などの実装用語は記載しない**
  - 例：❌ `approval_status=WAITING`、`ApprovalProcess.create(category=REQUEST)`、`is_invoice_approval`
  - 例：⭕ 「承認状況を『承認待ち』に更新」「承認処理履歴を『承認依頼』として登録」「請求最終承認権限を保有」
- **例外**：「10. テーブル概要」のみ物理テーブル名・カラム名を記載してよい（対応表として必要なため）
  - テーブル概要では **テーブルごとに小項目（H3 `### ■ {テーブル名}`）として切り出し、テーブル単位の表**で記述する（テーブル名を1列目に並べた横断表にしない）
- **APIパス・HTTPメソッド・ファイルパス・コード行番号**も記載しない（詳細設計・API設計書で定義）
- **状態・操作種別・権限**などは **日本語の業務用語** で表現する

```text
要件定義
  ↓
① 機能仕様書（テンプレートから作成開始）
   ↓ 画面遷移の記述が必要になったら
   ├─ WF（画面遷移・画面要素）を作成 → 機能仕様書に反映   ※ 画面を伴わない機能はスキップ可
   ↓ 画面概要の記述が必要になったら
   ├─ 画面概要（画面名・目的）を記述 → 機能仕様書に反映
   ↓ 画面遷移の記述が必要になったら
   ├─ 画面遷移図（mermaid）を記述 → 機能仕様書に反映
   ↓ 処理フローの記述が必要になったら
   ├─ シーケンス図を作成 → 機能仕様書に反映
   ├─ 処理フロー・処理ロジック詳細を記述 → 機能仕様書に反映
   ↓ APIの記述が必要になったら
   ├─ API概要を作成 → 機能仕様書に反映
   ↓ データ構造の記述が必要になったら
   └─ テーブル概要を作成 → 機能仕様書に反映
```

## 3. ■作業手順（Issueのタスクリストとして利用）

- [ ] **処理概要の記述**：[01_処理概要.md](../01_処理概要.md) に目的・背景を記述します。
- [ ] **アクターの記述**：[02_アクター.md](../02_アクター.md) に担当者・システムの役割を記述します。
- [ ] **ワークフローの記述**：[03_ワークフロー.md](../03_ワークフロー.md) に業務の状態遷移・進行を記述します。（画面を伴わない機能はスキップ）
- [ ] **WF（画面要素）の記述**：[03_ワークフロー.md](../03_ワークフロー.md) の画面要素テーブルに各画面のUI要素を記述します。[WF手順書](../11_WF.md) を参照します。（画面を伴わない機能はスキップ）
- [ ] **画面概要の記述**：[04_画面概要.md](../04_画面概要.md) に各画面の目的を業務レベルで記述します。（画面を伴わない機能はスキップ）
- [ ] **画面遷移図の記述**：[05_画面遷移図.md](../05_画面遷移図.md) に画面間の遷移関係を mermaid で記述します。（画面を伴わない機能はスキップ）
- [ ] **シーケンス図の記述**：[06_シーケンス図.md](../06_シーケンス図.md) に正常系・異常系の処理フローを記述します。
- [ ] **処理フローの記述**：[07_処理フロー.md](../07_処理フロー.md) に正常系のステップを番号で記述し、各ステップのエラーハンドリングをインデントで追記します。処理ロジック詳細が必要なステップには `（詳細は8.x参照）` を付記します。
- [ ] **処理ロジック詳細の記述**：[08_処理ロジック詳細.md](../08_処理ロジック詳細.md) にバリデーション条件（8.1）・登録内容（8.2）・処理制御（8.3）を記述し、処理フローとの対応を明示します。操作種別が複数ある場合はバリデーション条件を `#### ▶ 8.1.x 操作名（7.x）` の小項目に分割し、処理フロー側では `（詳細は 8.1.x 参照）` で紐づけます。
- [ ] **API概要の記述**：[09_API概要.md](../09_API概要.md) に利用するAPIの役割と概要を記述します。
- [ ] **テーブル概要の記述**：[10_テーブル概要.md](../10_テーブル概要.md) に対象テーブルとカラムをテーブルごとの小項目形式で記述します。
- [ ] **整合性の確認**：各セクションファイルの記述に矛盾がないかを確認します。

## 4. ■セルフチェック項目（作業完了後の最終確認）

- [ ] **「10. テーブル概要」以外に変数名・カラム名・クラス名・enum名が記載されていない**
- [ ] APIパス・HTTPメソッド・ファイル参照が記載されていない（詳細設計で定義）
- [ ] 状態・操作種別・権限などが日本語の業務用語で表現されている
- [ ] 機能のスコープ・アクター・目的が明記されている
- [ ] 処理フローの正常系ステップが記述されており、各ステップのエラーハンドリングがインデントで明示されている
- [ ] 処理フローと処理ロジック詳細（バリデーション条件・登録内容・処理制御）の対応が明記されている
- [ ] WF・シーケンス図・API概要・テーブル定義との内容が一致している
- [ ] 更新対象のテーブル・カラムが1行1カラム形式で特定されている
- [ ] ステータス管理がある場合、その遷移条件が網羅されている
- [ ] 不要な項目および他ドキュメントからの重複記載を避けている
- [ ] 用語が用語集・共通ルールに準拠している
- [ ] エラーメッセージ方針（文言・表示位置）と矛盾がない
- [ ] 第三者が読んで実装とテストの着手が可能な状態になっている
- [ ] AIによる自動生成内容を人間がレビューし、必要に応じて修正・補足されている
- [ ] ユーザー担当者・PMが見て内容を理解できる状態になっている
