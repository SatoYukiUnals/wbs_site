# DB設計規約

テーブル・カラム・マイグレーションの命名と設計ルールを定める。実装とすり合わせ時に不整合があれば本ファイルを更新する。

## 1. 基本方針

- **RDBMS**：プロジェクトで指定のもの（PostgreSQL / MySQL 等）に準拠する
- **ORM**：Django ORM を使用する。生SQLは原則禁止（[コーディング規約_Django](./02_コーディング規約_Django.md) 参照）
- **正規化** を基本とする。パフォーマンス要件で非正規化する場合はPRで理由を明記する
- **論理削除** を原則とする（物理削除はバッチ等で明示的に実施）
- **文字コード**：`UTF-8`。照合順序はプロジェクト指定（一般に `utf8mb4_unicode_ci` / `C.UTF-8` 等）

## 2. 命名規則

### ■ テーブル名

- **snake_case**・**単数形** を使用する
- 関連テーブル（中間テーブル）は `{テーブルA}_{テーブルB}_relation` 形式

| 区分 | 例 |
| :--- | :--- |
| OK | `contract`、`partner`、`broadcast_group` |
| OK | `broadcast_group_partner_responsible_person_relation`（中間テーブル） |
| NG | `contracts`（複数形） |
| NG | `Contract`（PascalCase・先頭大文字） |
| NG | `tbl_contract`（プレフィックス `tbl_` 不要） |

### ■ カラム名

- **snake_case** を使用する
- 命名は **意味が伝わる名詞** を選ぶ（略語を多用しない）
- 真偽値は **`is_` / `has_` / `can_`** プレフィックス
- 日時は **`_at`**、日付は **`_on`** / **`_date`** サフィックス

| 区分 | 例 |
| :--- | :--- |
| OK | `order_no`、`customer_id`、`is_active`、`created_at`、`order_date` |
| NG | `orderNo`（キャメルケース） |
| NG | `flg_active`（`flg` / `flag` は使わず `is_`）|
| NG | `nm`（`name` に） |

### ■ 主キー

- **`id`** を主キーとする
- 型は **`BIGINT`（自動採番）** を基本とする（小規模テーブルは `INT` 可）
- UUIDを使う場合は個別レビューで合意する

### ■ 外部キー

- **`{参照先テーブル名}_id`** の形式
- 参照制約（`ON DELETE` / `ON UPDATE`）をマイグレーションで明示する
- Django の `models.ForeignKey` 使用時は `on_delete` を必ず指定

| 区分 | 例 |
| :--- | :--- |
| OK | `customer_id`（`customer` テーブルへの参照） |
| OK | `created_by`（ユーザー参照。意味を込めた命名の場合） |
| NG | `customerId`、`customerRef` |

## 3. 共通カラム（監査列）

すべてのテーブルに以下のカラムを持たせる（Django の抽象基底モデルで統一）。

| カラム名 | 型 | NULL | デフォルト | 内容 |
| :--- | :--- | :---: | :--- | :--- |
| `id` | `BIGINT` | - | auto | 主キー |
| `created_at` | `DATETIME` | - | 現在日時 | 登録日時 |
| `updated_at` | `DATETIME` | - | 現在日時 | 更新日時（更新時に自動更新） |
| `created_by` | `BIGINT` | ✅ | `-` | 登録ユーザーID |
| `updated_by` | `BIGINT` | ✅ | `-` | 更新ユーザーID |
| `deleted_at` | `DATETIME` | ✅ | `NULL` | 論理削除日時（未削除は `NULL`） |

- 論理削除は **`deleted_at IS NULL`** を生存判定に使う
- バッチ・システム実行時の `created_by` / `updated_by` はシステムユーザーIDを入れる

## 4. データ型

### ■ 基本型

| 用途 | 型 | 備考 |
| :--- | :--- | :--- |
| 整数（主キー・外部キー） | `BIGINT` | 9桁以上を想定 |
| 整数（小規模） | `INT` | 上限が明確な場合 |
| 金額 | `DECIMAL(14, 2)` | 小数誤差を避けるため `FLOAT` は使わない |
| 文字列（短） | `VARCHAR(N)` | 上限を必ず指定 |
| 文字列（長） | `TEXT` | 200文字を超える可能性がある場合 |
| 真偽値 | `BOOLEAN` | Django では `BooleanField` |
| 日付 | `DATE` | `YYYY-MM-DD` |
| 日時 | `DATETIME` | タイムゾーンはUTC保存・JST表示（[共通ルール.md]で別途定義） |
| 列挙 | `VARCHAR(N)` | DBのENUM型は避け、文字列で管理 |
| JSON | `JSON` / `JSONB` | 構造化データ。頻繁に絞り込む項目は正規化する |

### ■ 文字列長の目安

| 用途 | 長さ |
| :--- | :--- |
| 氏名・名称 | `VARCHAR(100)` |
| 長い名称・備考 | `VARCHAR(200)` 〜 `TEXT` |
| メールアドレス | `VARCHAR(254)` |
| 電話番号 | `VARCHAR(20)` |
| 郵便番号 | `VARCHAR(10)` |
| ステータス・区分 | `VARCHAR(20)` |
| コード・番号 | `VARCHAR(12)` 〜 `VARCHAR(32)` |

## 5. NULL / NOT NULL

- **原則 NOT NULL** とし、NULLを許容する場合はビジネス的な意味を明確にする
- 未入力は `NULL` ではなく、**空文字 `""` や `-`** を使う場合もある。ただしプロジェクト内で統一する
- 真偽値は NULL を許容しない（3値になるのを避ける）

## 6. インデックス

- **主キー**：自動で作成される
- **外部キー**：必ずインデックスを張る
- **検索条件によく使うカラム**：単一・複合インデックスを検討する
- **ユニーク制約**：業務上一意であるべき項目は UNIQUE を付ける（`order_no` 等）
- 命名規則：
  - 通常インデックス：`idx_{table}_{columns}`（例：`idx_order_customer_id_status`）
  - ユニークインデックス：`uk_{table}_{columns}`

## 7. 制約

| 制約 | 使用方針 |
| :--- | :--- |
| PRIMARY KEY | 全テーブル必須（`id`） |
| FOREIGN KEY | 参照整合性を担保。`on_delete` を必ず指定 |
| UNIQUE | 業務キーに付与 |
| NOT NULL | 原則すべてのカラム |
| CHECK | 値の範囲や正規表現。DBに依存しやすいので Django バリデータ優先 |
| DEFAULT | 既定値がある場合に設定 |

## 8. マイグレーション

### ■ 命名規則

Django の標準形式 `{番号4桁}_{動作}_{対象}.py` に従う。

| 例 | 意味 |
| :--- | :--- |
| `0001_initial.py` | 初期作成 |
| `0042_add_status_to_contract.py` | `contract` に `status` を追加 |
| `0043_alter_contract_status.py` | `contract.status` の変更 |
| `0044_remove_contract_memo.py` | `contract.memo` を削除 |

### ■ マイグレーション作成時の注意

- **1マイグレーション = 1論理変更** を推奨
- 大量データを伴う変更は、**スキーマ変更** と **データ移行** を分けたマイグレーションにする
- **本番環境で取り返しのつかない変更**（カラム削除・型変更）はリリース計画に含めてレビューする
- Django のマイグレーションファイルは必ず Git 管理下に置く（`--fake` 使用時は慎重に）

### ■ インデックス追加

- 本番環境では **CONCURRENTLY**（PostgreSQL）/ **ONLINE**（MySQL）等、ロックを最小化する手段を用いる
- 大規模テーブルへのインデックス追加は、メンテナンス枠で実施する

## 9. バージョン管理・テーブル定義書

- テーブル定義書は `02_データ・共通設計/` 配下で管理する
- マイグレーションと定義書が **二重管理** にならないよう、定義書を正として追従するかマイグレーションを正とするかをプロジェクトで決める
- カラム追加・削除時は定義書も更新する（PRで差分が見えるように）

## 10. 禁止事項

- テーブル名に複数形（`contracts`）を使用する
- `tbl_` / `m_` などのプレフィックス
- `flg` / `flag` を真偽値名として使う
- `FLOAT` で金額を保存する
- DBの `ENUM` 型を使う（値追加時のマイグレーションが重くなるため）
- 生SQL・ストアドプロシージャ（特別な理由がない限り）
- 物理削除（論理削除が標準）
- 外部キー制約の省略（パフォーマンス理由以外）
- インデックスのないカラムでの頻繁な検索
- マイグレーション外でのスキーマ変更（直接DBを叩く）
- 個人情報・認証情報の平文保存（ハッシュ化・暗号化する）
