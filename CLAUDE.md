## 3. ドキュメント間の参照関係

```
要件定義書
    │
    ↓（機能要件を具体化）
基本設計：機能仕様（12機能） + システム構成図・DB設計・画面一覧・画面遷移図・APIエンドポイント一覧
    │
    ↓（各項目を詳細化）
詳細設計：API設計 / 画面仕様書 / IF仕様書 / バッチ仕様書 / 結合テスト / 総合テスト
    │
    ↓（設計を実装に落とし込む）
実装：DB / API / フロントエンド / バッチ処理
```

新たに機能追加・仕様変更を行う場合は、要件定義 → 基本設計 → 詳細設計 → 実装 の順で、上流から整合性を保ちながら更新すること。

## 4. 作業時のルール

### 4.1 ドキュメント更新ルール

- 既存ドキュメントを編集するときは、対応するフェーズ成果物（要件定義 / 基本設計 / 詳細設計）のバージョン・最終更新日も併せて更新する。
- 進捗ステータスが変わった場合は [進捗.md](進捗.md) も必ず更新する。
- 新規ドキュメントを作成する場合は、必ず `docx/00_ルール/` 配下の該当フェーズのテンプレート・手順書・サンプルに従うこと。

### 4.2 ファイル命名規約

- **機能仕様書**：`<番号2桁>_<機能名>.md` 例：`04_WBS・タスク管理.md`
- **API詳細設計**（予定）：`<パス区切りを_にしたもの>_<メソッド>.md` 例：`projects_project_id_tasks_POST.md`
- **画面仕様書**（予定）：`<画面ID>_<画面名>.md` 例：`04-00-00_WBS一覧画面.md`

画面ID・API URL体系などの詳細は、基本設計フェーズで作成する「画面一覧」「APIエンドポイント一覧」を参照する。

## 5. コード規約

### 5.1 コメント規約（現状の唯一の開発ルール）

- **コメントはすべて日本語で記述すること。** ソースコード（Python / JavaScript / TypeScript / Vue 等）・Markdown 内のコメント・コミットメッセージ内の補足など、すべてのコメントを日本語で統一する。
- 英語のコメントが既存コードに残っている場合は、編集時に日本語へ置き換える。
- 変数名・関数名・クラス名など **識別子は英語** のままとし、コメントのみ日本語とする。
