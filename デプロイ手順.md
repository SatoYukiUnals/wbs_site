## 0. 構成概要

本プロジェクトは以下の3コンテナ構成です（[docker-compose.prod.yml](docker-compose.prod.yml) 参照）。

| サービス名 | 役割 | ポート公開 |
|----------|------|----------|
| `db` | PostgreSQL 16（永続化ボリューム `postgres_data`） | なし（内部のみ） |
| `backend` | Django + Gunicorn（[backend/Dockerfile](backend/Dockerfile)） | なし（`frontend` 経由） |
| `frontend` | Vue3 ビルド済み + Nginx（リバースプロキシ兼静的配信、[frontend/nginx.conf](frontend/nginx.conf)） | **80** |

> Redis / Celery は後続フェーズで追加予定のため、本マニュアルでは扱いません。

```
クライアント
    │  HTTP(S)
    ▼
[frontend:80] ─ Nginx
    ├─ /         → Vue SPA（index.html フォールバック）
    ├─ /api/     → backend:8000（Gunicorn）
    ├─ /admin/   → backend:8000（Django 管理画面）
    ├─ /static/  → static_volume（collectstatic 成果物）
    └─ /media/   → media_volume（アップロードファイル）
            │
            ▼
       [backend:8000] ── DATABASE_URL ─→ [db:5432]
```

## 1. ConoHa VPS 契約（月額固定）

### 料金プラン（まとめトク）

月額固定で使うなら「まとめトク」を選びます。
契約期間が長いほど割引率が上がります。

| メモリ | 1ヶ月 | 12ヶ月 | 36ヶ月 | おすすめ用途 |
|--------|-------|--------|--------|------------|
| 1GB | 約550円 | 約440円 | 約380円 | 個人・小規模 |
| **2GB** | **約1,100円** | **約880円** | **約760円** | **常時稼働推奨** |
| 4GB | 約2,200円 | 約1,760円 | 約1,430円 | トラフィックが多い場合 |

> Django + Vue3 + Docker（PostgreSQL 同居）の構成なら **2GB を推奨**
> （1GB だと frontend ビルド時にスワップが発生して遅くなる。将来 Redis / Celery を追加するなら 4GB 視野）

1. https://vps.conoha.jp にアクセス

### 1-2. SSH キーの事前準備（WSL で先に作成）

サーバー追加前に SSH キーを作っておきます。

```bash
# WSL ターミナルで実行
ssh-keygen -t ed25519 -C "conoha-deploy"
# 保存先はデフォルト（~/.ssh/id_ed25519）のまま Enter × 2

# 公開鍵を表示してコピーしておく
cat ~/.ssh/id_ed25519.pub
# ssh-ed25519 AAAA... conoha-deploy ← これをコピー
```

### 1-3. サーバー追加

1. 管理画面の左メニューから「サーバー追加」をクリック
2. 以下の設定を選択

| 項目 | 設定値 | 備考 |
|------|--------|------|
| サービス | VPS | |
| 料金タイプ | **まとめトク** | 月額固定 |
| 契約期間 | **12ヶ月** | 長期利用なら36ヶ月がお得 |
| リージョン | 東京 | |
| OS | Ubuntu 22.04 | |
| アプリケーション | **Docker** | Docker が最初から入った状態で作成される |
| メモリ | **2GB** | 常時稼働なら2GB推奨 |
| root パスワード | 任意で設定 | **必ず控えておく** |
| SSH Key | 「インポート」から公開鍵を貼り付け | 手順1-2 でコピーしたもの |

3. 「追加する」をクリック → 1〜2分でサーバーが起動する

### 1-4. IP アドレスの確認

管理画面 → サーバーを選択 → 「ネットワーク情報」の「IPアドレス」を控える

```
例: 123.456.789.0
```

### 1-5. セキュリティグループの設定

> ⚠️ ここを忘れると SSH も Web も繋がらないので必ず設定する

**① セキュリティグループを作成する**

1. 左メニューの「**セキュリティ**」をクリック
2. 「**セキュリティグループ**」タブを選択
3. 右上の「**+ セキュリティグループ追加**」をクリック
4. 名前を入力（例：`wbs-sg`）して「**追加**」

**② ルールを追加する（3回繰り返す）**

作成したグループ名をクリック → 「**ルール追加**」

以下を1つずつ追加します。

**SSH（22番）**

| 項目 | 設定値 |
|------|--------|
| 方向 | ingress（受信） |
| プロトコル | TCP |
| ポート範囲 | 22 |
| IP バージョン | IPv4 |
| CIDR | 0.0.0.0/0 |

**HTTP（80番）**

| 項目 | 設定値 |
|------|--------|
| 方向 | ingress |
| プロトコル | TCP |
| ポート範囲 | 80 |
| IP バージョン | IPv4 |
| CIDR | 0.0.0.0/0 |

**HTTPS（443番）**※ 独自ドメインで HTTPS 化する場合のみ必要

| 項目 | 設定値 |
|------|--------|
| 方向 | ingress |
| プロトコル | TCP |
| ポート範囲 | 443 |
| IP バージョン | IPv4 |
| CIDR | 0.0.0.0/0 |

**③ サーバーに適用する（これを忘れると意味がない）**

1. 左メニューの「**サーバー**」をクリック
2. 対象サーバーをクリック
3. 「**ネットワーク情報**」タブを選択
4. 「**セキュリティグループ**」の「**編集**」をクリック
5. 作成したグループ（`wbs-sg`）を選択して「**保存**」

**④ 確認（WSL から）**

```bash
ssh -i ~/.ssh/id_ed25519 root@123.456.789.0
# "Welcome to Ubuntu" が出れば OK
```

**よくある失敗**

```
× ルールを追加したがサーバーへの適用を忘れた → SSH が繋がらない
× CIDR を空白のまま保存した               → 全拒否になる
× ingress ではなく egress を設定した       → 外からアクセスできない
```

### 2-5. 本番用 .env ファイルの作成

> **作業場所：VPS 上（SSH 接続した状態で実行）**

#### ① `.env` を作成する

`docker-compose.prod.yml` は `${POSTGRES_*}` などの変数をプロジェクトルートの `.env` から自動で読み込みます。

```bash
# VPS 上で実行
cd /var/www/wbs_site

# .env.example をコピー
cp .env.example .env

# vi で編集（後述の内容を入力）
vi .env
```

> vi の操作方法：
> - `i` キー → 編集モードに入る
> - 編集が終わったら `Esc` → `:wq` → `Enter` で保存して終了

#### ② `.env` の内容（[.env.example](.env.example) ベース）

```env
# ── PostgreSQL 設定 ──────────────────────────────
POSTGRES_DB=wbs_db
POSTGRES_USER=wbs_user
POSTGRES_PASSWORD=強いパスワードに変更すること

# ── Django 設定 ──────────────────────────────────
# 必ずランダムな文字列に変更（生成方法は ③）
DJANGO_SECRET_KEY=ここにランダムな文字列を入れる

# 本番アクセスに使う IP / ドメインをカンマ区切りで列挙
ALLOWED_HOSTS=123.456.789.0

# CORS 許可オリジン（フロントが同一オリジン配信なら IP/ドメインだけで OK）
CORS_ALLOWED_ORIGINS=http://123.456.789.0

# ── Gunicorn 設定 ────────────────────────────────
# (CPU コア数 × 2) + 1 が目安。2GB プランなら 3 で十分
GUNICORN_WORKERS=3
```

> 補足：
> - `DATABASE_URL` は [docker-compose.prod.yml](docker-compose.prod.yml) 内で `POSTGRES_*` から組み立てているため `.env` に書く必要はありません。
> - 独自ドメインで HTTPS 化する場合は `ALLOWED_HOSTS=example.com,123.456.789.0` のようにドメインも追加します。
> - `CORS_ALLOWED_ORIGINS` は frontend と backend を同一オリジン（同じ Nginx）で配信する本構成では実質不要ですが、settings.py が読み込むため空でもダミー値を入れておきます。

#### ③ `DJANGO_SECRET_KEY` の生成方法

WSL 側で生成してコピーします。

```bash
# WSL ターミナルで実行（Django がインストール済みの環境であれば）
python3 -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"

# Django が無ければこれでも可
python3 -c "import secrets; print(secrets.token_urlsafe(50))"
```

#### ④ Django 設定との対応（実作業手順）

「`.env` のどの値が、どのファイルのどの行を経由して、Django の `DATABASES` 設定にたどり着くのか」を、**実際に開いて確認するファイル**と**走らせるコマンド**の単位で示します。

##### Step 1. `.env`（VPS：`/var/www/wbs_site/.env`）に DB 用の値を書く

> 編集対象：`/var/www/wbs_site/.env`（このファイルだけ。後述の `settings.py` や `docker-compose.prod.yml` には触らない）

`.env.example` をコピーした直後の `.env` に、以下4行が入っていることを確認します。**`DATABASE_URL=` は書かない**（compose 側で組み立てるため）。

```env
POSTGRES_DB=wbs_db
POSTGRES_USER=wbs_user
POSTGRES_PASSWORD=<強いパスワード>
DJANGO_SECRET_KEY=<③で生成したランダム文字列>
```

確認コマンド（VPS 上）：

```bash
cd /var/www/wbs_site
grep -E '^(POSTGRES_|DJANGO_SECRET_KEY)=' .env
# 上記4行が表示されれば OK
grep -E '^DATABASE_URL=' .env && echo "⚠ DATABASE_URL が書かれている — 削除してください"
```

##### Step 2. `docker-compose.prod.yml` で値が組み立てられることを確認する（編集不要）

> 確認対象：[docker-compose.prod.yml](docker-compose.prod.yml)（既に正しく書かれているため**編集不要**。新規にゼロから書く場合のみ以下を追加する）

| 行番号 | 何が起きるか |
|---|---|
| [L19-L22](docker-compose.prod.yml#L19-L22) | `db` サービスに `POSTGRES_DB` / `POSTGRES_USER` / `POSTGRES_PASSWORD` を渡し、PostgreSQL の初期 DB・ロールを作る |
| [L44](docker-compose.prod.yml#L44) | `backend` サービスに `DATABASE_URL=postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB}` を組み立てて渡す |
| [L55-L57](docker-compose.prod.yml#L55-L57) | `depends_on: condition: service_healthy` で、`db` が `pg_isready` で応答するまで `backend` の起動を待つ |

新規にゼロから書く場合に `backend` サービス内で必須となる行：

```yaml
backend:
  environment:
    DATABASE_URL: postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB}
    SECRET_KEY: ${DJANGO_SECRET_KEY}
    DEBUG: "False"
    ALLOWED_HOSTS: ${ALLOWED_HOSTS}
  depends_on:
    db:
      condition: service_healthy
```

確認コマンド（VPS 上、コンテナ起動後）：

```bash
# backend コンテナに実際に渡された DATABASE_URL を覗く
docker compose -f docker-compose.prod.yml exec backend printenv DATABASE_URL
# → postgresql://wbs_user:<パスワード>@db:5432/wbs_db が出れば OK
```

##### Step 3. `backend/config/settings.py` の DB 接続コードを確認する（編集不要）

> 確認対象：[backend/config/settings.py](backend/config/settings.py) の **L74-L93**（既に書かれているため**編集不要**。新規にゼロから書く場合のみ以下を追加する）

```python
# settings.py L74-L93（このまま入っていることを確認）
_database_url = config('DATABASE_URL', default='')

if _database_url:
    import dj_database_url
    DATABASES = {
        'default': dj_database_url.parse(_database_url, conn_max_age=600)
    }
else:
    DATABASES = {
        'default': {
            'ENGINE': 'django.db.backends.postgresql',
            'NAME': config('DB_NAME', default='wbs_db'),
            'USER': config('DB_USER', default='wbs_user'),
            'PASSWORD': config('DB_PASSWORD', default='wbs_password'),
            'HOST': config('DB_HOST', default='db'),
            'PORT': config('DB_PORT', default='5432'),
        }
    }
```

`backend/requirements.txt` に以下が入っていることも確認：

```bash
# WSL or VPS のリポジトリルートで
grep -E '^(python-decouple|dj-database-url|psycopg)' backend/requirements.txt
# python-decouple / dj-database-url / psycopg2-binary（または psycopg）の3つが必要
```

##### Step 4. 値が流れたか実機で確認する（コマンド3本）

VPS 上で `docker compose up -d` 済みの状態で、以下を順に実行：

```bash
cd /var/www/wbs_site

# (1) compose が .env を展開して backend に DATABASE_URL を渡したか
docker compose -f docker-compose.prod.yml exec backend printenv DATABASE_URL
# 期待: postgresql://wbs_user:****@db:5432/wbs_db

# (2) Django 側で DATABASES['default'] が正しく組み上がったか
docker compose -f docker-compose.prod.yml exec backend \
    python manage.py shell -c "from django.conf import settings; import json; print(json.dumps({k: v for k, v in settings.DATABASES['default'].items() if k != 'PASSWORD'}, indent=2, default=str))"
# 期待: ENGINE=django.db.backends.postgresql / NAME=wbs_db / USER=wbs_user / HOST=db / PORT=5432

# (3) 実際に DB へ接続できるか（migrate を打つのが一番確実）
docker compose -f docker-compose.prod.yml exec backend python manage.py migrate --plan
# 期待: マイグレーション一覧が表示される（接続エラーが出ない）
```

##### まとめ：何をどこに置くか

| やること | 対象ファイル | 編集の要否 |
|---|---|---|
| DB 名・ユーザ・パスワード・SECRET_KEY を定義 | `/var/www/wbs_site/.env`（VPS） | **新規作成**（`.env.example` をコピーして値を埋める） |
| `DATABASE_URL` を `${POSTGRES_*}` から組み立てて backend へ注入 | [docker-compose.prod.yml](docker-compose.prod.yml) L44 | 既存のまま（**編集不要**） |

| `.env` を Git に上げない | [.gitignore](.gitignore) | 既存のまま（**編集不要**、`.env` 登録済み） |

## 4. Basic 認証の設定

本プロジェクトでは **既存の `frontend` コンテナ（Nginx 同梱）** に Basic 認証を組み込みます。
新たに別の Nginx コンテナを建てる必要はありません。

### 4-1. .htpasswd の生成（WSL で実行）

```bash
# apache2-utils をインストール（未インストールの場合）
sudo apt install apache2-utils -y

# .htpasswd を frontend ディレクトリに生成（ユーザー名は任意）
htpasswd -c frontend/.htpasswd <your-username>
# → パスワードの入力を求められる（2回入力）
```

### 4-2. .gitignore に追加（重要）

```bash
echo "frontend/.htpasswd" >> .gitignore
```

### 4-3. [frontend/nginx.conf](frontend/nginx.conf) の修正

`server { ... }` ブロックの先頭付近に Basic 認証の指示を追加します。
**ただし `/healthz` だけは認証対象から外す**（コンテナのヘルスチェックなどに使うため）。

```nginx
server {
    listen 80;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    # === Basic 認証（全パスに適用、/healthz のみ除外） ===
    auth_basic            "Restricted Area";
    auth_basic_user_file  /etc/nginx/.htpasswd;

    # セキュリティヘッダー
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options SAMEORIGIN;
    add_header X-XSS-Protection "1; mode=block";

    # （以下、既存の /api/ /admin/ /static/ /media/ / の各 location はそのまま）

    # ヘルスチェック用エンドポイント（Basic 認証なし）
    location /healthz {
        auth_basic off;
        access_log off;
        return 200 "ok";
        add_header Content-Type text/plain;
    }
}
```

### 4-4. [frontend/Dockerfile](frontend/Dockerfile) で .htpasswd をコンテナへ取り込む

`frontend/Dockerfile` の最終ステージ（Nginx を起動する側）に以下の行を追加します。

```dockerfile
# Basic 認証用のパスワードファイル
COPY .htpasswd /etc/nginx/.htpasswd
```

> Dockerfile を編集したくない場合は、代わりに [docker-compose.prod.yml](docker-compose.prod.yml) の `frontend` サービスに次のボリュームマウントを追加する方法でも可。
> ```yaml
> frontend:
>   volumes:
>     - ./frontend/.htpasswd:/etc/nginx/.htpasswd:ro
> ```

### 4-5. 本番反映

```bash
# WSL で deploy.sh を実行（.htpasswd を VPS に転送 → ビルド → 起動）
./deploy.sh main
```

### 4-6. パスワードを変更したいとき

```bash
htpasswd -c frontend/.htpasswd <your-username>
./deploy.sh main
```

### 4-7. ユーザーを追加したいとき

```bash
# -c をつけると上書き、つけないと追記
htpasswd frontend/.htpasswd <another-user>
./deploy.sh main
```

## 5. デプロイスクリプトの設定

リポジトリルートに `deploy.sh` を作成します。
本プロジェクトのサービス名（`db` / `backend` / `frontend`）に合わせています。

```bash
#!/usr/bin/env bash
# =====================================================================
# WBS管理ソフト デプロイスクリプト
#   ./deploy.sh                       # main を本番にデプロイ
#   ./deploy.sh feature/xxx           # 任意ブランチをデプロイ
#   ./deploy.sh feature/xxx --reset   # DB を破棄してから再構築
#   ./deploy.sh feature/xxx --reset --seed seeds/demo.py  # 初期データ投入
# =====================================================================
set -euo pipefail

# ─────────────────────────────────────────────────────────────
# 設定（環境に合わせて書き換える）
# ─────────────────────────────────────────────────────────────
SSH_HOST="root@123.456.789.0"        # ConoHa の IP
SSH_KEY="$HOME/.ssh/id_ed25519"      # WSL の秘密鍵パス
REMOTE_DIR="/var/www/wbs_site"       # VPS 上のプロジェクトパス
COMPOSE_FILE="docker-compose.prod.yml"
DB_SERVICE="db"
BACKEND_SERVICE="backend"
FRONTEND_SERVICE="frontend"
HTPASSWD_LOCAL="./frontend/.htpasswd"
HTPASSWD_REMOTE="frontend/.htpasswd"

# ─────────────────────────────────────────────────────────────
# 引数パース
# ─────────────────────────────────────────────────────────────
BRANCH="${1:-main}"
shift || true

RESET=0
SEED_FILE=""
while [[ $# -gt 0 ]]; do
    case "$1" in
        --reset) RESET=1; shift ;;
        --seed)  SEED_FILE="$2"; shift 2 ;;
        *) echo "unknown option: $1"; exit 1 ;;
    esac
done

ssh_run() {
    ssh -i "$SSH_KEY" "$SSH_HOST" "cd $REMOTE_DIR && $1"
}

# ─────────────────────────────────────────────────────────────
# ① .htpasswd を VPS にアップロード（存在する場合のみ）
# ─────────────────────────────────────────────────────────────
if [[ -f "$HTPASSWD_LOCAL" ]]; then
    echo "[1/6] Uploading .htpasswd"
    scp -i "$SSH_KEY" "$HTPASSWD_LOCAL" "$SSH_HOST:$REMOTE_DIR/$HTPASSWD_REMOTE"
else
    echo "[1/6] .htpasswd not found locally — skipping"
fi

# ─────────────────────────────────────────────────────────────
# ② ブランチ取得・切り替え
# ─────────────────────────────────────────────────────────────
echo "[2/6] git fetch & checkout $BRANCH"
ssh_run "git fetch --all --prune && git checkout $BRANCH && git pull origin $BRANCH"

# ─────────────────────────────────────────────────────────────
# ③ DB リセット（--reset 指定時）
# ─────────────────────────────────────────────────────────────
if [[ "$RESET" -eq 1 ]]; then
    echo "[3/6] Resetting database volume"
    ssh_run "docker compose -f $COMPOSE_FILE down -v"
else
    echo "[3/6] Skip DB reset"
fi

# ─────────────────────────────────────────────────────────────
# ④ ビルド & 起動
# ─────────────────────────────────────────────────────────────
echo "[4/6] Build & up"
ssh_run "docker compose -f $COMPOSE_FILE up -d --build"

# ─────────────────────────────────────────────────────────────
# ⑤ マイグレーション + collectstatic
# ─────────────────────────────────────────────────────────────
echo "[5/6] migrate & collectstatic"
ssh_run "docker compose -f $COMPOSE_FILE exec -T $BACKEND_SERVICE python manage.py migrate --no-input"
ssh_run "docker compose -f $COMPOSE_FILE exec -T $BACKEND_SERVICE python manage.py collectstatic --no-input"

# ─────────────────────────────────────────────────────────────
# ⑥ シードデータ投入（--seed 指定時）
# ─────────────────────────────────────────────────────────────
if [[ -n "$SEED_FILE" ]]; then
    echo "[6/6] seeding from $SEED_FILE"
    ssh_run "docker compose -f $COMPOSE_FILE exec -T $BACKEND_SERVICE python manage.py shell < $SEED_FILE"
else
    echo "[6/6] Skip seed"
fi

echo "✅ Deploy completed: branch=$BRANCH reset=$RESET seed=${SEED_FILE:-none}"
```

> 補足：
> - `entrypoint.sh` がコンテナ起動時に自動で `migrate` を流すため、上記 `[5/6]` は冗長ですが、ブランチ切替直後の追従用として残しています。
> - シードコマンドは Django 管理コマンドが実装され次第 `python manage.py loaddata fixtures/demo.json` などに置き換えてください。

## 6. デプロイの実行

```bash
# 通常デプロイ（main ブランチ）
./deploy.sh

# ブランチ指定
./deploy.sh feature/new-ui

# DB リセット付き（モデル変更・ブランチ大幅変更時）
./deploy.sh feature/new-ui --reset

./deploy.sh feature/new-ui --reset --seed seeds/demo.py
```

### デプロイの流れ

```
① .htpasswd を VPS にアップロード（存在時のみ）
② ブランチ取得・切り替え
③ DB リセット（--reset 指定時のみ）
④ docker compose build + up
⑤ migrate + collectstatic
⑥ シードデータ投入（--seed 指定時のみ）
```

## 7. 管理マニュアル

### コンテナの状態確認

```bash
# SSH でサーバーに接続
ssh -i ~/.ssh/id_ed25519 root@123.456.789.0
cd /var/www/wbs_site

# 全コンテナの状態確認
docker compose -f docker-compose.prod.yml ps

# コンテナを停止
docker compose -f docker-compose.prod.yml down

# コンテナを再起動
docker compose -f docker-compose.prod.yml restart
```

### ログの確認

```bash
# 全サービスのログ
docker compose -f docker-compose.prod.yml logs

# Django のログ（リアルタイム）
docker compose -f docker-compose.prod.yml logs -f backend

# Nginx（フロント）のログ
docker compose -f docker-compose.prod.yml logs -f frontend

# PostgreSQL のログ
docker compose -f docker-compose.prod.yml logs -f db

# 直近100行だけ
docker compose -f docker-compose.prod.yml logs --tail=100 backend
```

### DB の操作

```bash
# Django shell
docker compose -f docker-compose.prod.yml exec backend python manage.py shell

# PostgreSQL に直接接続
docker compose -f docker-compose.prod.yml exec db psql -U wbs_user -d wbs_db

# マイグレーション状態の確認
docker compose -f docker-compose.prod.yml exec backend python manage.py showmigrations

# スーパーユーザー作成（初回のみ）
docker compose -f docker-compose.prod.yml exec backend python manage.py createsuperuser
```

### バックアップ（PostgreSQL）

```bash
# DB をホストにダンプ
docker compose -f docker-compose.prod.yml exec -T db \
    pg_dump -U wbs_user wbs_db > /var/www/wbs_site/backup_$(date +%Y%m%d).sql

# 復元
cat backup_YYYYMMDD.sql | docker compose -f docker-compose.prod.yml exec -T db \
    psql -U wbs_user -d wbs_db
```

### ブランチ切り替えパターン

```bash
# コードだけ変わった（モデル変更なし）
./deploy.sh feature/ui-fix

# モデルが変わった（マイグレーションで対応可能）
./deploy.sh feature/add-profile

# モデルが大きく変わった or 旧ブランチに戻す
./deploy.sh feature/xxx --reset

# 全く別のブランチでクリーン環境から
./deploy.sh feature/other-project --reset --seed seeds/demo.py
```

## 8. トラブルシューティング

### SSH に繋がらない

```bash
# セキュリティグループでポート22が開いているか確認
# 管理画面 → セキュリティ → ポート22が許可されているか

# 秘密鍵の権限を修正
chmod 600 ~/.ssh/id_ed25519
```

### docker compose が失敗する

```bash
# エラーログを確認
docker compose -f docker-compose.prod.yml logs backend

# 全部落として再ビルド
docker compose -f docker-compose.prod.yml down
docker compose -f docker-compose.prod.yml up -d --build
```

### migrate でエラーが出る

```bash
# マイグレーション状態を確認
docker compose -f docker-compose.prod.yml exec backend python manage.py showmigrations

# DB リセットして最初からやり直す
./deploy.sh <current-branch> --reset
```

### 502 Bad Gateway が出る

```bash
# backend が起動しているか確認
docker compose -f docker-compose.prod.yml ps

# .env の ALLOWED_HOSTS に VPS の IP / ドメインが含まれているか確認
grep ALLOWED_HOSTS .env

# staticfiles を再収集
docker compose -f docker-compose.prod.yml exec backend python manage.py collectstatic --no-input
```

### Basic 認証ダイアログが出ない

```bash
# .htpasswd が VPS の frontend コンテナに反映されているか確認
ssh -i ~/.ssh/id_ed25519 root@123.456.789.0
ls /var/www/wbs_site/frontend/.htpasswd
docker compose -f /var/www/wbs_site/docker-compose.prod.yml exec frontend ls /etc/nginx/.htpasswd

# 反映されていなければ deploy.sh を再実行
./deploy.sh main
```

### `frontend/nginx.conf` の変更が反映されない

`frontend` イメージは Dockerfile で nginx.conf を焼き込むため、**設定変更後は必ず再ビルドが必要**です。

```bash
docker compose -f docker-compose.prod.yml up -d --build frontend
```

## まとめ：帰宅後の作業順序

```
① WSL で SSH キーを生成（ssh-keygen）
     ↓
② ConoHa でアカウント作成
     ↓
③ まとめトク・2GB・Docker テンプレートでサーバー追加
   （SSH Key に公開鍵を貼り付け）
     ↓
④ セキュリティグループでポート 22・80（必要なら 443）を開放
     ↓
⑤ SSH で VPS に接続 → /var/www/wbs_site にクローン → .env を作成
     ↓
⑥ WSL で frontend/.htpasswd を生成
   ＋ frontend/nginx.conf に Basic 認証を追記
   ＋ frontend/Dockerfile に COPY .htpasswd /etc/nginx/.htpasswd を追記
     ↓
⑦ deploy.sh の IP・パスを書き換え
     ↓
⑧ ./deploy.sh で初回デプロイ
     ↓
⑨ ブラウザで http://<IP>/ にアクセス → Basic 認証ダイアログが出れば完了
```
