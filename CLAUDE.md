## 3. ID体系・命名規約

### 3.1 画面ID

`[カテゴリ番号]-[データ単位番号]-[操作区分]` の3階層で構成。

- カテゴリ番号：`01`=認証 / `02`=プロジェクト / `03`=クォーター / `04`=WBS / `05`=ガント / `06`=ロードマップ / `07`=ダッシュボード / `08`=レビュー / `09`=報告書 / `10`=Excel / `11`=テンプレート
- 操作区分：`00`=一覧 / `01`=登録 / `02`=変更 / `03`=詳細 / `04以降`=その他

詳細は [docx/02_基本設計/画面一覧.md](docx/02_基本設計/画面一覧.md) を参照。

### 3.2 API URLベース

`/api/v1` 配下に全エンドポイントを配置。認証は JWT Bearer Token。

### 3.3 ファイル命名

| 種別 | 規約 | 例 |
|------|------|----|
| API詳細 | `<パス区切りを`_`>_<メソッド>.md` | `projects_project_id_tasks_POST.md` |
| 画面仕様 | `<画面ID>_<画面名>.md` | `04-00-00_WBS一覧画面.md` |
| 単体テスト | `<画面ID>_<画面名>/` ディレクトリ | `04-00-00_WBS一覧画面/` |
| 総合テスト | `SCN-<番号>_<シナリオ名>.md` | `SCN-01_テナント登録からプロジェクト作成まで.md` |

## 4. ドキュメント間の参照関係

```
要件定義書
    │
    ↓（機能要件を具体化）
基本設計：機能仕様（12機能）
    │        ├─→ 画面一覧・画面遷移図
    │        ├─→ APIエンドポイント一覧
    │        ├─→ DB設計
    │        └─→ システム構成図
    │
    ↓（各項目を詳細化）
詳細設計：API設計 / 画面仕様書 / IF仕様書 / 単体テスト / 総合テスト
```

新たに機能追加・仕様変更を行う場合は、要件定義 → 基本設計（機能仕様＋関連ドキュメント）→ 詳細設計 の順で上流から整合性を保つこと。

---

## 5. 作業時のルール

- 既存ドキュメントを編集するときは対応する成果物（要件定義 / 基本設計 / 詳細設計）のバージョン・最終更新日も併せて更新する。
- `機能仕様_原案/` は履歴参照用のため、仕様変更時は `機能仕様/` 側のみ更新する。
- 進捗ステータス変更時は [進捗管理.md](進捗管理.md) も更新する。
- ファイルを新規作成する場合、本ファイルの「3. ID体系・命名規約」に従うこと。

## 6. コード規約

### 6.1 コメント

- **コメントはすべて日本語で記述すること。** ソースコード（Python / JavaScript / TypeScript / Vue 等）・Markdown 内のコメント・コミットメッセージ内の補足など、すべてのコメントを日本語で統一する。
- 英語のコメントが既存コードに残っている場合は、編集時に日本語へ置き換える。
- 変数名・関数名・クラス名など**識別子は英語**のままとし、コメントのみ日本語とする。
