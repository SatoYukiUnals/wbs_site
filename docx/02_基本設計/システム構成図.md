## 2. 認証フロー（JWT）

```mermaid
sequenceDiagram
    autonumber
    participant Client as クライアント (Vue.js)
    participant API as バックエンド (Django)
    participant Redis as Redis

    Client->>API: POST /auth/login/ {email, password}
    API->>API: パスワード検証
    API->>Redis: リフレッシュトークン保存
    API-->>Client: {access_token（15分）, refresh_token（7日）}

    Note over Client,API: 通常リクエスト
    Client->>API: GET /projects/ Authorization: Bearer {access_token}
    API->>API: アクセストークン検証
    API-->>Client: 200 OK {data}

    Note over Client,API: トークンリフレッシュ
    Client->>API: POST /auth/refresh/ {refresh_token}
    API->>Redis: リフレッシュトークン検証
    API-->>Client: {access_token（新規）}

    Note over Client,API: ログアウト
    Client->>API: POST /auth/logout/ {refresh_token}
    API->>Redis: リフレッシュトークン削除（無効化）
    API-->>Client: 200 OK
```

## 4. マルチテナント方式

- テナント分離方式：**共有スキーマ方式**（全テナントが同一DBスキーマを利用）
- 全テーブルに `tenant_id` を持たせ、APIリクエスト処理時にJWTから `tenant_id` を取得してクエリフィルタリング
- テナント間のデータ混在をバックエンドのORMレイヤーで制御

```mermaid
graph LR
    subgraph Tenants["複数テナント"]
        T1["テナントA"]
        T2["テナントB"]
        T3["テナントC"]
    end
    subgraph App["共有アプリケーション"]
        API["Django API"]
        DB[("PostgreSQL\n（共有スキーマ）")]
    end
    T1 -->|"JWT(tenant_id=1)"| API
    T2 -->|"JWT(tenant_id=2)"| API
    T3 -->|"JWT(tenant_id=3)"| API
    API -->|"WHERE tenant_id=N"| DB
```
