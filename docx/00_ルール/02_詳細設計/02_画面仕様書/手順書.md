# 詳細設計：画面仕様書

## 1. ■概要・目的

Frontend の実装に必要な詳細情報を定義します。各項目のバリデーションルール、エラー表示、状態管理、API呼び出しのタイミング等を具体化することで、要件通りの画面動作を実現します。

## 2. ■事前準備

- [ ] 規定のテンプレート：[仕様書_template.md](./仕様書_template.md) をコピーして作業を開始する
- [ ] 項目説明テンプレート：[項目説明_template.md](./項目説明_template.md)（エンドユーザー向けマニュアル用）
- [ ] 画面のWF・デザインプロトタイプ（ある場合）
- [ ] 基本設計のテーブル定義・API概要

本仕様書は2つのドキュメントで構成する：

1. **画面仕様書** ([仕様書_template.md](./仕様書_template.md) / [仕様書_sample.md](./仕様書_sample.md))：実装者向けの技術仕様
2. **項目説明** ([項目説明_template.md](./項目説明_template.md) / [項目説明_sample.md](./項目説明_sample.md))：エンドユーザー・PM向けのマニュアル

## 3. ■作業手順（Issueのタスクリストとして利用）

画面仕様書は以下の順序で構成する。状態管理は該当がない場合も項目を残し「なし」と記載する。

1. 基本情報
2. 目的
3. 処理概要
4. アクセス権限
5. ダイアログ
6. 機能仕様
7. バリデーション
8. アクション
9. 状態管理
10. 参照テーブル
11. 更新テーブル

- [ ] **基本情報**：以下の項目を記載する
  - 画面ID
  - 画面名
  - URL
  - 対応デザイン（デザインファイルへのリンク）
  - ファイル命名規則：`画面ID_画面名.md`（例：`SC-08-01_配信先グループ登録.md`）
  - 画像命名規則：`画面ID_画面名_01.png`。パターンが複数ある場合は `_01`、`_02` と連番にする

- [ ] **目的**：この画面で何を行うのかを1〜2文で記載する

- [ ] **処理概要**：以下の内容を中項目（H3）で記載する
  - 画面で操作できる情報（入力・選択できる情報）
  - 画面の遷移（遷移元・遷移先）

- [ ] **アクセス権限**：この画面にアクセスするために必要な権限を記載する。権限制御がない場合は記載不要

- [ ] **ダイアログ**：この画面で使用するダイアログの一覧を記載する（ない場合は「なし」）

- [ ] **機能仕様**：以下の構成で記載する

  - **項目一覧**：画面に表示されるすべての要素を表で記載する
    - 項目名：画面に表示されているラベル名を使用する
    - data-testid：テストで要素を識別するための id。実装コードを確認して記載する
    - 型：「テキスト」「テキストボックス」「テキストエリア」「セレクトボックス」「ラジオボタン」「チェックボックス」「日付」「ボタン」「画像」などから選択する
    - 繰り返し：繰り返しがある場合は数字を記載する。1 始まりで、別の繰り返し配列には +1 した番号を割り振る
    - 初期状態：「活性」「非活性」「非表示」から選択する
    - 文字揃え：「左」「中央」「右」から選択する

  - **備考**：項目一覧の直下に配置する。特記事項がある項目のみ記載する

  - **入力フィールド**：入力系の項目について以下を記載する
    - 項目名・必須・型・初期値・データ型
    - 必須は `✅`、任意は `-`。初期値が未入力の場合は `-`（「空」は使わない）

  - **出力フィールド**：表示系の項目について以下を記載する
    - 項目名・内容・初期値

  - **静的フィールド**：APIに依存しない固定表示の項目を記載する（ない場合は「なし」）

  - **ボタン**：ボタン名と押下時の挙動を記載する

- [ ] **バリデーション**：項目ごとに H3 で区切り、以下を記載する
  - トリガー（「入力確定時」「ボタン押下時」など）
  - チェック・追加ルール・エラーID・エラーメッセージの表

- [ ] **アクション**：トリガーごとに H3 で区切り、処理内容を箇条書きで記載する
  - 見出しは `### ■ ボタン名（押下時）` の形式にする（「クリック時」は使わない）
  - 使用するAPI・遷移先があれば合わせて記載する

- [ ] **状態管理**：可変のデータによって表示制御が行われる場合に定義する。該当がない場合は「なし」と記載する
  - 各 State について、型・初期値・変更トリガー・用途を記載する

- [ ] **参照テーブル**：出力フィールドの参照元テーブル・カラムを記載する
  - テーブル名・カラム名はバッククォートで囲む。データ型は大文字（`VARCHAR`、`BIGINT` 等）

- [ ] **更新テーブル**：入力フィールドの更新先テーブル・カラムを記載する
  - テーブル名・カラム名はバッククォートで囲む。データ型は大文字（`VARCHAR`、`BIGINT` 等）

- [ ] **項目説明の作成**：[項目説明_template.md](./項目説明_template.md) をコピーして、エンドユーザー向けのマニュアル形式で項目説明を記載する
  - ファイル名：`画面ID_画面名_項目説明.md`
  - 画面上のグループ・エリア単位でセクションを区切る（①、②…）
  - 必須項目には `❗` をつける
  - 技術用語を避け、利用者の操作視点で記述する
  - 補足は `💡`、注意は `⚠️` のマーカーを使い、blockquote で強調する

## 4. ■セルフチェック項目（作業完了後の最終確認）

- [ ] 基本情報（画面ID・URL・対応デザイン）が記載されている
- [ ] 処理概要に「画面で操作できる情報」「画面の遷移」が記載されている
- [ ] 項目一覧にすべての画面要素が漏れなく記載されている
- [ ] 入力フィールドのすべての項目にバリデーションが定義されている
- [ ] バリデーションにトリガー・チェック・エラーIDが記載されている
- [ ] アクションに使用APIと遷移先が記載されている
- [ ] 状態管理が必要な場合は定義されており、不要な場合は「なし」と記載されている
- [ ] 参照テーブル・更新テーブルが記載されている
- [ ] 項目説明（エンドユーザー向けマニュアル）が作成されており、画面仕様書の項目と整合が取れている
- [ ] テンプレートの `> [記]` で始まる行をすべて削除している
- [ ] 不要な項目、および他ドキュメントから変更のない重複記載を避けている
- [ ] AIによる自動生成内容を人間がレビューし、必要に応じて修正・補足されている
- [ ] ユーザー（角田さん）、PMが見てロジックや実装方針を理解できる内容になっている
